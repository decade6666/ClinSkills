# review-changes

对 git 暂存区（staged）或全部未提交修改（unstaged）进行系统评估，
用 markdown 表格输出结果和优化建议。

## 工作流程

### 1. 确定检查范围

默认检查**暂存区**（`git diff --cached`），因为这是用户准备提交的改动。
如果暂存区为空但工作区有修改，则检查**工作区**（`git diff`）。
用户可通过自然语言指定范围（"全查"/"只查暂存"/"只查某个文件"）。

```bash
git diff --cached --stat          # 了解改动了哪些文件
git diff --cached                 # 完整 diff（输出长时分批读取）
```

**快速模式判定**：`--stat` 显示仅 1 个文件且为 `.py`、变更行数 ≤ 20 行时，自动启用快速模式：
- 跳过维度 2-6（矛盾、过度设计、流程影响、更好策略、关键遗漏——小改动几乎不触发）
- 仅执行：spawn `python-reviewer` Agent（`mode: diff`）→ 并入输出表格
- 输出概览中加一行：`评估模式: 快速（仅 Python 规范 + 明显错误，已跳过维度 2-6）`
- 非 Python 文件或超过阈值 → 走完整 7 维度流程

### 2. 理解改动上下文

对每个变更文件：
- 新增文件：`Read` 全文
- 修改文件：读变更行及周围 30 行上下文 + 文件头部（导入、配置）
- 删除文件：读 git 中删除的关键行（`git diff --cached -- <file>`）

同时检查 `.claude/`, `config.py`, `config.yaml`, `CLAUDE.md` 等全局配置文件是否有相关约定，避免误判为「违反规范」而实际是遵循约定。

### 3. 按 7 维度逐项评估

对每个变更，对照以下 7 个维度逐一判断。**某维度无问题则跳过，不在输出表格中占行。**

**Agent 辅助：** 变更涉及 Python 文件时，并行 spawn `python-reviewer` Agent，prompt 中指定 `mode: diff`，对 diff 做 PEP 8 / Pythonic + 明显错误深度审查，覆盖维度 1 和维度 7。Agent 返回结果并入输出表格（其 致命/重要/建议 三级映射为本表的 严重/高/中/低，归入维度 1/7），主 skill 不再重复检查这两个维度的 Python 专项问题，聚焦维度 2-6（项目感知维度）。非 Python 文件（.md / .yaml / .json）全部维度由主 skill 人工审查。

| # | 维度 | 核心判据 | 审查方式 |
|---|------|---------|---------|
| 1 | **最佳实践** | 命名（变量/文件/函数）、导入方式、项目规范（CLAUDE.md / coding-guide / 语言惯例）、安全实践（无密钥泄露、输入校验） | Python→Agent / 其他→人工 |
| 2 | **矛盾** | 新增/修改代码与同一 diff 中其他文件的约定是否冲突；与现有代码/配置/文档是否矛盾 | 人工 |
| 3 | **过度设计** | 是否存在未请求的功能、不必要的抽象层、一次性的工具函数、未被调用但已引入的依赖 | 人工 |
| 4 | **流程影响** | 是否改变了数据流方向、文件路径约定、导入结构、CLI 入口、或上游依赖此文件的脚本 | 人工 |
| 5 | **更好策略** | 是否有更简洁的写法（表达式而非循环）、更高效的数据结构、可复用的已有 utils 函数、或更适合此场景的标准库 | 人工 |
| 6 | **关键遗漏** | 缺少错误处理、缺少边界条件、缺少文档/注释同步更新、缺少 .gitignore 条目、缺少类型标注的关键位置、修改后未同步更新的索引/目录/README | 人工 |
| 7 | **明显错误** | 语法错误、未定义的变量/函数引用、逻辑错误（条件取反、off-by-one）、类型错误、空值未防护、死代码 | Python→Agent / 其他→人工 |

### 4. 输出格式

**必须使用以下 Markdown 模板：**

```markdown
# 暂存区修改评估

## 概览
- 修改文件: N 个 （有新 / 修改 / 重命名 / 删除）
- 发现问题: N 项 （严重 N / 高 N / 中 N / 低 N）
- 评估范围: 暂存区（git diff --cached）

## 逐项评估

| # | 维度 | 文件 | 位置 | 问题描述 | 严重度 | 优化建议 |
|---|------|------|------|---------|--------|---------|
| 1 | 明显错误 | foo.py | L42 | `df[col]` 前未判空，col 可能不在 columns 中 | 严重 | 加 `if col not in df.columns: continue` |
| 2 | 过度设计 | bar.py | L15-30 | 5 行逻辑写了 15 行包装函数，仅单处调用 | 低 | 内联到调用处或确认有复用需求再抽 |
| ... | ... | ... | ... | ... | ... | ... |

## 优化建议汇总

1. **严重**（必须修复）：...
2. **高**（建议修复）：...
3. **中**（建议考虑）：...
4. **低**（可选）：...
```

**严重度判定：**

| 严重度 | 含义 |
|--------|------|
| 严重 | 运行时报错、数据丢失、安全漏洞、破坏已有流程 |
| 高 | 功能缺陷、显著影响可维护性、与项目规范严重冲突 |
| 中 | 代码质量问题、不一致、可能埋雷 |
| 低 | 风格/惯例/可读性小瑕疵 |

**表格约束：**
- 每条问题独占一行
- 同一文件-维度-行号的同类问题可合并为一行
- 位置写精确行号（单行为 `L34`，范围为 `L15-30`，跨文件为 `foo.py:L12`）
- 问题描述写「当前状态」而非「应该怎样」（后者在建议列）
- 某维度全局无问题时，概览中注明「无」
- **不在表格中输出空行或无问题的维度**

### 5. 输出后行为

- 不主动提交或修改代码——仅输出评估表格。
- 如用户追问某条建议，再做针对性修改。
- 如全部维度无问题（罕见），输出「所有修改经 7 维度检查未发现明显问题。」并逐项标注「通过」。
