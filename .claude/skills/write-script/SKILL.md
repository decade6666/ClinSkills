# write-script

根据用户需求，查询元数据，编写符合项目规范的 Python 数据核查脚本。

## 工作流程

### 0. 确认源数据表头结构

**优先读取 CLAUDE.md**：先查看 `CLAUDE.md` 的 Conventions 中是否已记录 `表头结构`（`header` / `skiprows`）及列名语言。有则直接沿用，无需再问。

没有记录时，用 AskUserQuestion 向用户确认：

- **标题行数**：Excel 前几行是列名行（非数据）？常见的有 1 行（单行表头）、2 行（双行表头，一行英文一行中文）
- **英文列名行**：第几行是英文列名？（1-indexed，如"第 2 行"）
- **pandas 使用哪行作为表头**：通常用英文 SAS 列名行（`header=0`），另一行用 `skiprows` 跳过

> **各 EDC 系统示例**：
> - **clinflash**：单行中文列名（`受试者编号`、`采样时间点(MITPT)`…）→ `header=0`（无 skiprows）
> - **cmis**：第 1 行英文 SAS 列名（SUBJID, VISIT, VISDAT…），第 2 行中文列名 → `header=0, skiprows=[1]`
> 不同 EDC 系统的行顺序可能不同，以实际数据为准。

确认后**立即写入 `CLAUDE.md` 的 Conventions 节**，如果不存在CLAUDE.md，直接创建。表头约定和列名语言因 EDC 类型而异，格式如下：

```markdown
# clinflash 示例
- 表头结构：`header=0`（单行中文列名，无 skiprows）

# taimei5/taimei6/cmis 示例
- 表头结构：`header=0, skiprows=[1]`（第 1 行英文 SAS 列名，第 2 行中文列名被跳过）
```

这样后续脚本无需重复询问。

### 1. 理解需求

需求来源可能是：
- 用户口述（如"我需要一个不良事件汇总表"）
- 参考已有脚本（如"按清单01的格式做一个新的"）
- 用户给出输出结果/图表示例

**关键动作：** 确认输出形状（三线表 docx 还是 Excel 清单）、涉及哪些数据域、筛选条件是什么。

任何时候需求不够清晰，直接用 AskUserQuestion 向用户提问。常见需要澄清的点：
- 输出是 docx 三线表还是 xlsx 清单
- 筛选条件（全量还是某个子集）
- 多列信息如何合并（取首/末、拼接、优先级）
- 合计行/百分比计算规则

### 2. 查询元数据

用 `query_metadata.py` 定位所需表和字段。路径相对于项目根目录：

```bash
python .claude/skills/write-script/scripts/query_metadata.py <command> [args]
```

命令清单与用法以脚本为准——**不带参数运行可打印完整命令列表**。常用命令速览：

| 命令 | 用途 |
|------|------|
| `summary` | 元数据概览（表单数 / 字段数 / 编码表数） |
| `search <关键字>` | 跨表搜索字段（按需求关键字定位候选） |
| `fields <表单名>` | 查看某表单全部字段（格式、编码表） |
| `codelist <名称>` | 查看编码表枚举值 |
| `find-field <SAS名>` | 按 SAS 字段名定位所在表单 |
| `field-codelist <字段名>` | 按字段名直接查其编码表枚举值 |
| `forms` / `codelists` / `visits` | 列出全部表单 / 编码表 / 访视 |

**查询策略（渐进式披露）：**

> 原则：用 `query_metadata.py` 按需逐条查询，**不要一次性读取整个 JSON 文件**。
> 先粗后细：`search` 定位 → `fields` 确认 → `codelist` 取值，每步只获取当前需要的信息。

1. 先 `search` 需求中的关键字，定位候选字段
2. 再 `fields <表单名>` 确认字段详情（格式、编码表）
3. 涉及编码值时 `codelist <名称>` 查看枚举
4. 确认哪些字段需要解码列（选项型字段需读 `itemName` + 解码后缀）

`fields` 命令的输出已包含解码列名（标注 `← 用此列`），**直接使用即可，无需回退读 Excel 表头验证**。只有当元数据中找不到某表单或字段时，才回退到直接读 Excel。

**解码后缀：** 带 `codeList` 的字段一定有两列——码值列和解码列。脚本中始终用解码列。`fields` 命令会根据本项目 EDC 系统（元数据 `_meta.edcType`）自动给出正确的解码列名（标注 `← 用此列`），**直接采用即可，无需手动套后缀**。各系统后缀对照（仅供理解，工具已自动处理）：

| EDC 系统 | 解码列格式 | 示例 |
|---|---|---|
| 太美5 / 太美6 | `_TXT` 后缀 | `临床评估_TXT` |
| 赛美斯（cmis） | `_DEC` 后缀 | `临床评估_DEC` |
| clinflash / 易迪希 | `{itemName}({fieldOID})`（含解码值，无独立后缀） | `临床评估(MIPERF)` |

**注意：** EDC 的 sheet 名就是表单 OID（如 `DS_END`、`EC_ED`、`SV`），不是中文表单名。`formOID` 字段就是 `load_sheet` 的第一个参数。

**元数据字段说明（FormField.json）：**

| 字段 | 含义 | 脚本中的用途 |
|---|---|---|
| `formOID` | 表单 OID = Excel sheet 名 | `load_sheet()` 第一个参数 |
| `formName` | 表单中文名 | 查询时定位表单 |
| `fieldOID` | 字段 OID（clinflash 用于构造列名） | clinflash: 列名为 `{itemName}({fieldOID})` |
| `sasFieldName` | SAS 字段名（英文） | SAS 到 pandas 的映射参考 |
| `itemName` | 字段中文标签 = Excel 列名 | `IMPORT_*` / `load_sheet` 的 `cols` 参数 |
| `fieldFormat` | 字段类型（下拉框 / 水平单选框 / 日期框 等） | 判断是否需要解码列 |
| `codeList` | 编码表引用（name / count） | `codelist` 命令查枚举值 |
| `codeList.hasOther` | 是否含"其他"自由文本选项 | 必须读解码列，码值列只有编码 |

### 3. 方案二次确认（动手前必须）

> ⚠️ **抗衰减规则：无论本会话已执行了多少个脚本、无论用户回复多么简短（"继续""好""OK"），此步骤不可跳过。确认必须基于两次表格呈现 + 用户明确肯定词，不得默认通过。**

查完元数据后已有大体思路，**先不要写代码**。先用 Markdown 表格把方案呈现给用户，等用户确认后再动手。

需确认的内容用两张表列清：

**① 用到的表与字段：**

| 表单 OID（sheet 名） | 字段（中文列名 `itemName`） | 用途 |
|---|---|---|
| `AE` | 不良事件名称 | 输出列 |
| `AE` | 严重程度（解码列 `严重程度_TXT`） | 筛选 + 输出列 |
| … | … | … |

**② 输出结果列：**

| No. | 输出列名 | 来源 / 计算规则 |
|---|---|---|
| 1 | 受试者编号 | DM.受试者筛选号 |
| 2 | 不良事件名称 | AE.不良事件名称 |
| … | … | … |

筛选条件、合并/计算规则等也一并列出，请用户一并确认。

**确认循环：**

- 用户回复含肯定词（"好""可以""确认""没问题""行"等）或直接要求开始写 → 视为同意，进入 Step 4
- 用户回复含"不""修改""改一下""错了"等修改信号 → 更新表格，重新呈现，再次等待
- 同一方案修改超过 3 次仍未收到明确确认 → 按最新表格版本执行，并告知用户

### 4. 编写脚本

确认数据需求后，**在动手写代码之前**先做三项检查（4a→4b→4c）：

**4a 检查 utils/ 已有函数：** 读 `utils/loaders.py` 和 `utils/output_format.py`，看是否有可直接复用的函数。不重复造轮子。

**4b 检索脚本库（先查再写，避免从零开始）：** 读 `.claude/skills/write-script/library/INDEX.md`（轻量目录，唯一入口），按需求关键词命中「标签」列。命中 1–2 个候选后，`Read` 对应模板全文，判定：

- **命中可复用** → 复制模板到 `04 scripts/<章节>/`、改顶部「项目配置」块的业务字段（`# @config` 注释头列出了需改的项）、按需微调主体。**系统列（6 个定位角色）由模板顶部的 `system_cols()` 自动解析，勿手填、勿在主体硬编码。** 模板形态为「系统列块 + 项目配置块 + 真实 Python 主体」，复制后即可语法检查与运行。
- **无合适模板** → 不强行套用，记录未命中原因（供 Step 6 入库评估参考），按 `coding-guide.md` 从头写。

> 渐进式披露：**先读 INDEX.md 选候选，再读候选全文**，不要一次性读取整个 library 目录。模板本身不连数据，仅作 `ast.parse` 语法检查的参考样本。
>
> 系统列注册表：`utils/loaders.py` 的 `SYSTEM_COLUMNS` 按 EDC 类型登记系统列名。若当前 EDC 未登记，`system_cols()` 抛清晰错误——按提示在注册表补一行（同一 EDC 跨研究固定，一次性补全即可）。

**4c 检查是否应封装为通用工具：** 如果当前逻辑（如日期解析、特定域的筛选合并、格式转换）很可能在后续脚本中重复出现，先将其封装到 `utils/` 中再调用，而不是在脚本内写死。注意区分：**函数级**复用进 `utils/`（import 后调用）；**整脚本级**复用进脚本库（复制后改配置块）。

完成上述检查后，按编码规范编写。每次编写脚本前读取 `coding-guide.md`：

```bash
Read .claude/skills/write-script/reference/coding-guide.md
```

### 5. 运行验证

脚本编写或修改完成后，**必须实际运行**，不得跳过。

**正常路径（原始数据可用）：**

```bash
python 04 scripts/<章节>/<脚本名>.py
```

从项目根目录执行。检查：

1. **退出码为 0** — 有非零退出码即为失败
2. **输出文件已生成** — 检查 `output_path` 下对应路径的文件是否存在
3. **输出文件非空** — 文件大小 > 0

**降级路径（原始数据不可用）：**

```bash
python -c "import ast; ast.parse(open(r'04 scripts/<章节>/<脚本名>.py', encoding='utf-8').read()); print('语法检查通过')"
```

仅验证语法正确性，提醒用户数据层未验证。

**错误处理：**

- 能定位原因 → 修复后重跑，循环直到通过
- 无法判断修复方向（如数据层报错、不确定的业务逻辑歧义）→ **直接将完整报错输出给用户**，附上你的初步分析（如有），由用户指定下一步方向。不要猜测性修复

**验证通过后：** 提醒用户查看输出文件进行复核。

### 6. 入库评估（脚本跑通后）

脚本验证通过后，评估其是否值得沉淀进脚本库（`.claude/skills/write-script/library/`）：

**入库判据：**
- 逻辑跨项目通用——换变量名/配置块即能在别的项目复用（如「时间窗口重合」「不完整日期核查」）。
- 项目特定业务逻辑（如某方案独有的给药规则）**不入库**。

**仅提议，不自动执行：** 若判定可入库，向用户呈现：
1. 建议的模板文件名（动词短语小写下划线，如 `xxx_check.py`）
2. 抽出的「项目配置」块（哪些项应参数化）
3. `# @desc / @tags / @config` 注释头草稿
4. 待追加到 `library/INDEX.md` 的一行

等用户确认后，才写入 `library/` 并更新 `INDEX.md`。用户未确认前不自动落盘。
