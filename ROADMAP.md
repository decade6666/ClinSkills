# 迭代计划

## 1. Data Review Plan 解析与覆盖检查

新增一个 skill（如 `parse-drp`），将 Excel 格式的 Data Review Plan 解析为结构化 JSON，核心能力：

- **解析 DRP Excel**：提取每条核查规则的分类（如"实验室检查""不良事件""合并用药"等）、核查要点、筛选条件、输出字段
- **按分类聚合**：同一分类下的核查规则合并为一组，识别可共用数据源的规则，减少脚本数量
- **覆盖检查**：对比 DRP 中的核查要点和已有脚本产出，标识未覆盖的核查项，生成覆盖缺口报告

目标：一套 DRP → 一组 JSON → 按分类批量生成脚本，而非逐条手写。

## 2. 脚本模板库

扩充 `skills/write-script/library/`，将常见核查场景的脚本存为模板：

- **DMR 报告固定表格**：如受试者列表、方案偏离汇总、AE 汇总、合并用药汇总等固定格式的表格，存为标准模板
- **模板标注约定**：模板中可能因项目而异的地方（表单 OID、字段列名、筛选编码值、输出列头等）用 `# ── TODO: <说明> ──` 注释标注，Agent 在生成时自动识别并确认
- **模板索引**：维护 `library/INDEX.md`，每个模板附带一句话说明 + 适用 EDC 类型 + 关键可变量列表

目标：常见核查场景从"每次从头写"变为"选模板 → 确认变量 → 生成"。

## 3. 特殊字段合并规则

在 `review-checklist.md` 和 `coding-guide.md` 中增加专项说明，覆盖两类易出错的字段合并场景：

### 3.1 "若其他，请详述"字段合并

带 `hasOther` 的编码字段（如 `asOther` 标注），原始内容为编码值 + 自由文本，输出时需将自由文本拼接到编码值后：
- **示例**：`MHTERM` 编码字段 + `MHTERM_OTHER` 自由文本字段，输出时格式为 `"编码值（解码值）；自由文本"`（如 `"999（其他）；具体描述内容"`）
- **注意点**：
  - 合并前需判空——自由文本为空时不拼接
  - 码值为编码表内选项时不拼接（只有"其他"码值才拼接）
  - taimei/cmis 有独立后缀（`_TXT`/`_DEC`），clinflash 解码值在主列中

### 3.2 给药原因合并

合并用药（CM）表单中，给药原因字段常分散在多个字段中，需合并输出。具体哪些字段参与合并**必须通过 `query_metadata.py` 查询 CM 表单的 `fields` 元数据确认**，不可硬编码。

常见参与合并的字段类型（实际字段名以元数据为准）：
- **关联不良事件**：编码表通常为 AE 编号或 AE 名称
- **关联既往病史**：编码表通常为 MH 编号或病史名称
- **预防用药**：如预防用抗感染、预防过敏等
- **其他**：自由文本或其他编码值

合并注意点：
- 合并逻辑需在 `Derive` 步骤完成，输出列头统一为"给药原因"
- 多个字段均有值时，用分隔符拼接（如 `"适应症A；原因说明B"`）
- 编码字段需通过 `codelist` 命令确认编码表取值后再定筛选/拼接逻辑

## 4. 脚本/输出文件名与存储结构一致性

脚本文件名、输出文件名、存储访视三层命名保持一致，避免「脚本叫 A，产出叫 B，数据来自访视 C」的混乱。

### 4.1 命名一致规则

以 `04 scripts/` 下脚本文件名为基准，其对应的输出文件和依赖的访视节点（存储访视）应保持命名对应：

- **脚本文件名**：`<功能描述>.py`，如 `lb_check.py`
- **输出文件名**：与脚本完全同名（仅扩展名不同），如 `lb_check.py` → `lb_check.xlsx` / `lb_check.docx`，不加任何后缀
- **平铺对应**：脚本在 `04 scripts/` 下平铺（无子目录）时，输出也在 `03 output/` 下平铺；脚本有子目录分组时，输出按同样结构镜像

### 4.2 组织结构一致性

`03 output/` 整体镜像 `04 scripts/` 的目录结构：
- 脚本平铺 → 输出平铺；脚本分目录 → 输出分同样目录
- 按访视拆分时，子目录名对齐 `query_metadata.py visits` 返回的正式访视名
- 「脚本 → 输出 → 数据源」可追溯链路：从文件名直接找到对应脚本和数据源，无需额外索引

### 4.3 编写/审查时的检查项

- 脚本名和输出文件名中的关键词一致
- 存储访视名与 `visits` 元数据正式名称一致，不自行简写
- 组织结构变动时同步更新相关的 README 或索引
